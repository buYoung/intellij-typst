# Typst Formatting Feature Design Doc

## 1. Document Intent

이 문서는 Typst 포매터 기능의 사용자 경험, 보존 정책, 실패 정책, 코드 스타일 경계를 정하는 의사결정 출처다.

이 문서는 포매터 구현 절차나 파일별 작업 목록이 아니다. 다운스트림 구현 에이전트는 이 문서에서 “어떤 결과가 올바른 포맷인가”를 읽어야 한다.

---

## 2. Background / Problem

Typst 문서는 사람이 읽는 본문과 구조화된 코드 표현식이 섞인다. 일반 텍스트처럼만 편집하면 목록, 함수 호출, 블록, 배열, 수식, 주석의 간격과 줄바꿈이 파일마다 쉽게 달라진다.

IDE 사용자는 다른 언어와 마찬가지로 “코드 재정렬” 명령이 Typst 문서에서도 예측 가능하게 동작하기를 기대한다. 하지만 문서 언어는 프로그래밍 언어보다 원문 텍스트와 줄바꿈의 의미가 사용자 의도에 가깝기 때문에, 포매터가 과도하게 문서를 바꾸면 작성자의 문체와 문서 레이아웃을 훼손할 수 있다.

이 기능은 네이티브 Typst 문법 모델 위에서 보수적이고 반복 가능한 포맷 결과를 제공하기 위해 필요하다.

---

## 3. Feature Definition

```text
Typst Formatting is the native IntelliJ formatting and comment-toggle experience for Typst files.
```

### This feature is

- Typst 파일에서 IDE의 포맷 명령과 주석 토글이 예측 가능하게 동작하게 하는 기능이다.
- Typst 문법 구조를 기준으로 들여쓰기, 줄바꿈, 간격을 정하는 기능이다.
- 사용자의 본문 작성 의도를 최대한 보존하는 보수적 문서 편집 기능이다.

### This feature is not

- Typst 문서를 임의 스타일로 재작성하는 미화 도구가 아니다.
- 렌더링 결과가 같다는 이유만으로 원문 구조를 크게 바꾸는 기능이 아니다.
- 외부 포매터나 언어 서버에 문서를 넘기는 기능이 아니다.

---

## 4. Goals & Non-Goals

### Goals

- Typst 코드 표현식, 블록, 함수 호출, 배열, 딕셔너리, 수식의 구조적 들여쓰기를 제공한다.
- 마크업 본문의 자유로운 줄바꿈과 문체를 불필요하게 바꾸지 않는다.
- IDE 코드 스타일 설정과 호환되는 반복 가능한 결과를 만든다.
- 전체 파일 포맷과 선택 영역 포맷의 안전 범위를 구분한다.
- 포맷 결과는 가능한 한 최소 편집으로 적용한다.
- 주석 토글이 v1에서 Typst 줄 주석 중심으로 예측 가능하게 동작한다.
- 불완전한 문서에서도 가능한 범위만 포맷하고 사용자 입력을 잃지 않는다.

### Non-Goals

- 렌더링 레이아웃을 기준으로 줄바꿈을 재계산하지 않는다.
- 문장 자동 줄바꿈이나 자연어 문단 정렬을 v1 목표로 삼지 않는다.
- 전체 Typst 컴파일러의 포맷 규칙과 완전 동일한 결과를 보장하지 않는다.
- 외부 명령 기반 포매터를 실행하지 않는다.

## 5. User Model & Core Concepts

### User Model

사용자는 포매터를 “코드처럼 구조가 있는 Typst 구간은 정리하지만, 내가 쓴 문서 본문은 함부로 바꾸지 않는 도구”로 이해한다.

사용자는 다음을 기대한다.

- 괄호, 블록, 함수 인자, 배열, 딕셔너리의 들여쓰기가 정돈된다.
- 주석 토글이 현재 선택 영역 또는 줄에 맞춰 자연스럽게 동작한다.
- 이미 잘 정리된 문서를 다시 포맷해도 결과가 흔들리지 않는다.
- 오류가 있는 문서에서도 전체 파일이 망가지지 않는다.

사용자가 이해하지 않아도 되는 것은 다음과 같다.

- 포매터 블록 모델
- IDE 코드 스타일 설정이 내부 간격 규칙으로 바뀌는 방식
- 구문 트리 복구 상태

### Core Concepts

| Concept | Meaning |
| ------- | ------- |
| 포맷 범위 | 사용자가 포맷을 요청한 전체 파일 또는 선택 영역 |
| 구조적 구간 | 들여쓰기와 간격을 안정적으로 정할 수 있는 Typst 코드 또는 수식 영역 |
| 자유 텍스트 구간 | 사용자의 문장, 줄바꿈, 문체를 우선 보존해야 하는 마크업 본문 |
| 안전 포맷 범위 | 구문적으로 닫혀 있고 원본 문서 범위로 되돌릴 수 있어 포맷 결과를 안정적으로 적용할 수 있는 범위 |
| 간격 정책 | 토큰 사이 공백과 줄바꿈을 결정하는 설계 규칙 |
| 주석 그룹 | 연속된 주석 줄이 하나의 보존 단위로 취급되는 범위 |
| 주석 토글 | IDE 명령으로 줄 주석을 추가하고 제거하는 동작 |

## 6. Relationship to Existing Features

| Existing Feature | Relationship |
| ---------------- | ------------ |
| Typst Native Language Services | 포매터가 신뢰할 Typst 문법 모델과 파일 인식을 제공한다. |
| Typst Syntax Highlighting | 강조와 포매터가 같은 문맥 전환과 토큰 분류를 공유해야 한다. |
| IDE Code Style | 사용자 선호 들여쓰기와 간격 설정을 해석하는 외부 사용자 설정으로 취급한다. |

## 7. Primary User Flows

### 7.1 Main Flow

```text
사용자가 .typ 파일에서 포맷 명령을 실행한다
  -> 시스템은 Typst 문법 구조와 사용자 코드 스타일 설정을 해석한다
  -> 구조적 구간은 정돈되고 자유 텍스트 구간은 보존된다
```

### 7.2 Secondary Flow

```text
사용자가 선택한 줄에서 주석 토글을 실행한다
  -> 시스템은 Typst 주석 문법에 맞게 주석을 추가하거나 제거한다
  -> 선택 영역과 커서 위치는 IDE 관례에 맞게 유지된다
```

### 7.3 Failure / Partial Success Flow

```text
문서에 닫히지 않은 괄호나 불완전한 코드 표현식이 있다
  -> 시스템은 안전하게 해석 가능한 구간만 포맷한다
  -> 해석할 수 없는 구간은 원문을 보존한다
```

## 8. Design

### 8.1 Behavior

- 포매터는 Typst 문법 구조를 기준으로 들여쓰기와 간격을 결정한다.
- 마크업 본문은 사용자의 줄바꿈과 공백 의도를 우선 보존하며, v1에서는 prose wrapping을 제공하지 않는다.
- 코드 표현식, 함수 호출, 함수 인자, 배열, 딕셔너리, 블록, 패턴, 이항 연산 체인, 수식 인자처럼 구조가 명확한 구간은 IDE 코드 스타일 설정에 맞춰 정렬한다.
- 마크업과 코드 또는 수식이 같은 줄에 섞인 경우에는 구조가 닫혀 있는 하위 범위만 정리하고, 불확실하면 원문 보존을 선택한다.
- 선택 영역 포맷은 선택한 문자 범위가 아니라 안전 포맷 범위를 기준으로 하며, 구조 중간만 선택된 경우 범위를 임의 확장하지 않고 원문 보존을 선택한다.
- 적용 결과는 전체 파일 재작성보다 문자 경계를 보존하는 최소 편집을 선호한다.
- 연속 주석은 보존 단위로 취급하고, 빈 줄, 문단 경계, 다른 토큰을 넘어 합치거나 재배치하지 않는다.
- 포맷 결과는 반복 적용해도 더 이상 바뀌지 않는 안정성을 가져야 한다.
- 주석 토글은 v1에서 Typst 줄 주석만 적용한다.

### 8.2 Conceptual Data Model

| Entity | Meaning |
| ------ | ------- |
| Format Region | 포맷 명령이 적용되는 문서 범위 |
| Safe Format Region | 포맷 결과를 원본 범위에 안정적으로 되돌릴 수 있는 구문 범위 |
| Format Block | 들여쓰기, 줄바꿈, 간격을 결정할 수 있는 문법 단위 |
| Preserved Text | 사용자의 원문 공백과 줄바꿈을 보존해야 하는 범위 |
| Spacing Rule | 두 문법 단위 사이 공백과 줄바꿈을 정하는 규칙 |
| Comment Group | 독립적으로 보존해야 하는 연속 주석 범위 |
| Comment Target | 주석 토글이 적용되는 줄 또는 선택 범위 |

| Field | Meaning |
| ----- | ------- |
| indent level | 구조적 중첩 깊이에 따른 들여쓰기 수준 |
| wrap behavior | 긴 구조를 줄바꿈할지 유지할지의 정책 |
| preservation mode | 원문 보존, 구조 정렬, 미해석 보존 중 하나의 처리 방식 |
| edit shape | 전체 교체, 선택 범위 교체, 최소 편집 중 사용자에게 적용되는 변경 형태 |
| comment kind | v1에서는 줄 주석으로 고정되는 주석 방식 |

### 8.3 Failure Handling

- 구문 트리가 불완전하면 안전하게 해석 가능한 하위 범위만 포맷한다.
- 자유 텍스트를 구조적 구간으로 오인할 위험이 있으면 원문 보존을 선택한다.
- 포맷 중 취소가 발생하면 사용자가 부분 수정으로 혼란을 겪지 않도록 IDE의 표준 쓰기 동작과 일관되어야 한다.
- 선택 영역이 함수 호출, 배열, 딕셔너리, 수식, 블록의 중간만 포함하면 임의로 상위 구조를 확장하지 않고 보존한다.
- 주석 토글 대상이 애매하면 줄 주석 적용도 보류하고 원문 보존을 우선한다.

## 9. Policy Decisions

### 9.1 보수적 포맷 정책

Decision:

- v1 포매터는 구조가 확실한 구간만 적극적으로 정리하고, 자유 텍스트는 보존한다.
- v1은 prose wrapping을 제공하지 않으며, 본문 줄바꿈은 사용자가 작성한 그대로 둔다.

Rationale:

- Typst는 문서 언어이므로 원문 줄바꿈과 공백이 작성자의 의도와 밀접하다.
- 과도한 자동 정리는 사용자가 포매터를 신뢰하지 못하게 만든다.

### 9.2 반복 안정성 정책

Decision:

- 같은 파일에 포맷을 반복 적용했을 때 결과가 계속 변하지 않아야 한다.
- 포맷 결과는 가능한 한 최소 편집으로 적용해 커서, 선택 영역, 버전 관리 차이를 안정적으로 유지한다.

Rationale:

- IDE 포매터는 저장 전 정리, 선택 영역 정리, 팀 코드 스타일 적용에 반복적으로 사용된다.
- 흔들리는 포맷은 불필요한 변경을 만들고 버전 관리 기록을 오염시킨다.

### 9.3 선택 범위 안전 정책

Decision:

- 선택 영역 포맷은 안전 포맷 범위 안에서만 적용한다.
- 선택 영역이 구조 중간을 자르면 상위 범위로 자동 확장하지 않고 원문 보존을 선택한다.

Rationale:

- 자동 범위 확장은 사용자가 선택하지 않은 문서까지 바꿀 수 있다.
- Typst 문서는 마크업, 코드, 수식이 섞이므로 구조 중간 포맷의 위험이 크다.

### 9.4 외부 포매터 배제 정책

Decision:

- 현재 범위에서는 외부 실행 파일이나 언어 서버 기반 포매팅을 사용하지 않는다.
- 기존 Typst 포매터 엔진 연동도 v1 범위에서 제외하고 IntelliJ 네이티브 포매터 경험을 우선한다.

Rationale:

- 사용자가 네이티브 IntelliJ API 기반 구현을 요청했다.
- 외부 포매터는 설치, 버전, 실행 실패, 보안 경계를 새로 만든다.

### 9.5 Typst 문법 기준선 정책

Decision:

- v1 포맷 기준선은 2026-06-30에 확인한 최신 `typst` crate 버전인 Typst 0.15.0 언어 표면에서 구조가 명확한 구문에 한정한다.
- 기준선 밖의 새 문법이나 애매한 구간은 원문 보존으로 처리한다.

Rationale:

- 포매터는 문서를 직접 바꾸므로 구문 강조보다 더 보수적인 기준선이 필요하다.
- 새 문법을 잘못 이해한 포맷은 사용자의 원문을 손상시킬 수 있다.

### 9.6 줄 주석 우선 정책

Decision:

- v1 주석 토글은 줄 주석만 지원한다.
- 블록 주석 토글은 명확한 사용자 요구와 안전한 범위 정책이 정해질 때까지 미래 확장으로 둔다.

Rationale:

- 줄 주석은 선택 영역과 줄 단위 편집 흐름에서 예측 가능하다.
- 블록 주석은 중첩, 선택 영역 경계, 원문 보존 정책과 충돌할 수 있어 별도 설계가 필요하다.

### 9.7 주석 그룹 보존 정책

Decision:

- 연속 주석은 보존 단위로 취급한다.
- 빈 줄, 문단 경계, 다른 토큰을 넘어 주석 그룹을 합치거나 재배치하지 않는다.
- 문서 주석과 일반 줄 주석은 표시와 보존에서 구분할 수 있지만, v1 주석 토글은 줄 주석 중심 정책을 유지한다.

Rationale:

- 주석은 코드 구조뿐 아니라 문서 설명과 작성자 의도를 담는다.
- 주석 그룹을 재배치하면 문서 의미가 바뀌거나 작성 흐름이 깨질 수 있다.

## 10. Alternatives Considered

### Alternative: 전체 파일 자동 재작성

Description:

- Typst 파일 전체를 하나의 강한 스타일 규칙으로 다시 쓴다.

Why not chosen:

- 자유 텍스트와 문서 작성 의도를 훼손할 가능성이 높다.
- 초기 문법 모델이 불완전한 상태에서 큰 범위 변경은 위험하다.

### Alternative: 외부 포매터 실행

Description:

- 별도 명령이나 언어 서버의 포맷 결과를 받아 문서에 적용한다.

Why not chosen:

- 네이티브 IntelliJ API 기반 구현이라는 제약과 맞지 않는다.
- 외부 도구 설치와 버전 차이가 사용자 경험을 불안정하게 만든다.

### Alternative: 포매터 없이 주석 토글만 제공

Description:

- 문서 구조 포맷은 제공하지 않고 주석 기능만 제공한다.

Why not chosen:

- IDE 사용자가 기대하는 기본 언어 편집 경험이 부족하다.
- 구조적 코드 구간 정리가 없으면 Typst 코드 표현식이 커질수록 편집 비용이 커진다.

---

## 11. Cross-cutting Concerns

### 11.1 Security

- 포매터는 로컬 문서 내용을 IDE 안에서 변환하며 외부 명령을 실행하지 않는다.
- 임의 파일 접근이나 네트워크 호출은 현재 범위에 포함하지 않는다.

### 11.2 Privacy

- 포맷 대상 문서 내용은 IDE 밖으로 전송되지 않는다.
- 포맷 결과나 문서 조각을 원격 수집하지 않는다.

### 11.3 Permissions

- 문서 수정은 사용자의 명시적 포맷 또는 주석 명령에 의해 발생한다.
- 변경은 IDE의 표준 실행 취소 모델과 호환되어야 한다.

### 11.4 Observability

- 개발 중에는 포맷 후에도 계속 바뀌는 영역, 보존 실패, 미해석 범위를 추적할 수 있어야 한다.
- 일반 사용 중에는 포매터 실패가 과도한 알림으로 표시되지 않아야 한다.

### 11.5 Accessibility

- 포매터와 주석 토글은 키보드 명령으로 접근 가능해야 한다.
- 포맷 실패나 제한은 IDE 표준 알림 또는 진단 표시와 호환되어야 한다.

### 11.6 Internationalization

- 사용자에게 보이는 포맷 오류와 설정 문구는 리소스 번들로 현지화할 수 있어야 한다.
- 다국어 본문과 유니코드 공백 처리는 원문 의미를 훼손하지 않아야 한다.

## 12. Scope

### In Scope for initial native Typst support (2026-06-30)

- 전체 파일 또는 선택 영역에 대한 보수적 Typst 포맷
- 안전 포맷 범위 안에서의 선택 영역 포맷
- 구조적 Typst 코드 및 수식 구간의 들여쓰기와 간격 정리
- 함수 인자, 배열, 딕셔너리, 블록, 패턴, 이항 연산 체인, 수식 인자의 구조 정리
- 마크업 본문과 미해석 구간의 원문 보존
- 최소 편집 적용
- 주석 그룹 보존
- 줄 주석 토글
- 반복 적용 안정성

### Out of Scope for initial native Typst support (2026-06-30)

- 렌더링 폭 기반 자동 줄바꿈
- 자연어 문단 재흐름
- prose wrapping 설정
- 블록 주석 토글
- 외부 포매터 실행
- 기존 Typst 포매터 엔진 연동
- 팀 공유 포맷 프로필 동기화
- 문서 전체 의미 분석에 의존하는 스타일 재작성

## 13. Risks & Open Questions

### Risks

- 문법 모델이 불완전하면 포매터가 구조적 구간과 자유 텍스트를 잘못 구분할 수 있다.
- 보수적 정책이 지나치면 사용자가 기대하는 정리 효과가 부족할 수 있다.
- 강한 포맷 정책을 너무 일찍 정하면 이후 Typst 문법 지원 확장과 충돌할 수 있다.
- 선택 영역 포맷을 보존 위주로 처리하면 사용자가 선택한 일부 구조가 바뀌지 않아 답답하게 느낄 수 있다.

### Open Questions

- Typst 표준 스타일과 프로젝트별 코드 스타일 설정이 충돌할 때 어느 쪽을 우선할 것인가?
- Typst 표준 스타일과 프로젝트별 코드 스타일 설정이 충돌할 때 사용자가 볼 설정 이름과 우선순위 표현을 어떻게 정할 것인가?

## 14. Platform Design

### 14.1 Common Design

포매터는 IntelliJ의 표준 포맷 명령과 주석 토글 경험에 통합되어야 한다. 사용자는 별도 도구를 실행한다고 느끼지 않고 기존 IDE 단축키와 메뉴를 그대로 사용한다.

### 14.2 IntelliJ Platform

문서 수정은 IDE의 실행 취소, 선택 영역, 코드 스타일 설정, 쓰기 동작과 일관되어야 한다. 포매터는 문법 단위 범위를 겹치지 않게 다루고, 불완전한 구문에서는 안전한 보존을 우선한다.

### 14.3 Remote Development

원격 개발 환경에서도 포맷은 사용자 명령에 대한 반응성을 유지해야 한다. 긴 파일 포맷은 취소 가능해야 하며, 원격 프론트엔드와 백엔드 사이에서 불필요한 대용량 왕복을 만들지 않는 것이 바람직하다.

## 15. Result Semantics

| State | Meaning | User-visible? |
| ----- | ------- | ------------- |
| Formatted | 요청 범위가 포맷 정책에 따라 정리되었다. | Yes |
| Preserved | 원문 보존 정책 때문에 변경하지 않았다. | Yes |
| Partially formatted | 해석 가능한 일부 범위만 정리되었다. | Yes |
| Minimal edit applied | 전체 재작성 대신 실제로 달라진 범위만 변경했다. | Yes |
| Comment toggled | 선택 영역이나 줄에 주석 토글이 적용되었다. | Yes |
| Unsupported | 현재 범위에서 안전하게 포맷할 수 없는 구조다. | Yes |
| Canceled | 사용자가 취소했거나 IDE가 작업을 중단했다. | Yes |
